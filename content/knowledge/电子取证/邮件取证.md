- 平台：Kali Linux / 通用 Linux
- 使用领域：Forensics / 邮件分析 / 钓鱼排查
- 主要用途：分析 `.eml` 邮件头、MIME 结构、正文、附件、链接与发信路径，适合钓鱼邮件、恶意附件、伪造发件人排查。
- 工具位置：`grep`、`sed`、`python3`、`file`、`strings`、`exiftool`、`ripmime/munpack(可选)`
- 当前状态：通用取证流程文档
- 适用对象：`.eml`、部分 `.msg` 转换后的邮件、导出的原始邮件源文件

## 邮件取证是做什么的

邮件取证的核心不是“看这封邮件写了什么”，而是判断：

- 这封邮件是谁发的，路径是否合理
- 发件人地址、显示名、回复地址是否一致
- 邮件正文里有没有钓鱼链接、伪装域名、社工内容
- 是否带了可疑附件、内嵌图片、压缩包、Office 文档、脚本
- SPF、DKIM、DMARC 是否通过
- 是否存在伪造头部、异常中继、可疑来源 IP

比赛、应急响应或日常安全排查里，`.eml` 通常比表面截图更有价值，因为原始头部、MIME 结构和附件编码都在里面。

## 一眼先看什么

拿到 `.eml` 后，建议先看这几项：

1. `From`、`Reply-To`、`Return-Path`
2. `Received` 链
3. `Authentication-Results`
4. `Subject`
5. `Content-Type`
6. 是否有附件、内嵌图片、HTML 正文

最简单的第一轮查看：

```bash
file mail.eml
head -n 80 mail.eml
grep -nE '^(From|To|Cc|Subject|Date|Reply-To|Return-Path|Message-Id|Authentication-Results|Received):' mail.eml
```

## 推荐分析顺序

建议按这个顺序做：

1. 看邮件头，判断发件链路和身份伪装
2. 看 MIME 结构，确认正文、附件、内嵌资源
3. 提取附件和正文
4. 分析 URL、域名、邮箱地址、IP
5. 对附件做 `file`、`hash`、`exiftool`、`strings`、杀软/YARA/沙箱分析
6. 回头把邮件头和正文线索串起来，判断是否钓鱼、诈骗、投递恶意载荷

## 最常用命令

### 1. 看核心邮件头

```bash
grep -nE '^(From|To|Cc|Subject|Date|Reply-To|Return-Path|Message-Id):' mail.eml
```

重点看：

- `From`：显示给收件人的发件人
- `Reply-To`：回复会发到哪里，钓鱼邮件经常这里不一致
- `Return-Path`：退信地址，能辅助看真实发件路径
- `Message-Id`：某些仿冒邮件会出现奇怪域名或生成器痕迹

### 2. 看认证结果

```bash
grep -nE '^Authentication-Results:|^Received-SPF:|^DKIM-Signature:' mail.eml
```

重点看：

- `spf=pass/fail`
- `dkim=pass/fail`
- `dmarc=pass/fail`

经验判断：

- 三者都通过，不代表一定安全，但伪造概率降低
- 如果 `From` 看起来像正规公司，但 `Reply-To`、`Return-Path` 或认证结果不一致，就要提高警惕

### 3. 看 Received 链

```bash
grep -n '^Received:' mail.eml
```

判断思路：

- 最底部的 `Received` 通常更接近源头
- 看是否出现家庭宽带 IP、异常国家/地区、云主机、动态拨号地址
- 看时间线是否连续、时区是否合理

### 4. 看 MIME 结构

```bash
grep -nE 'Content-Type:|Content-Disposition:|filename=|name=' mail.eml
```

常见情况：

- `multipart/mixed`：通常代表有附件
- `multipart/related`：HTML 正文和内嵌资源
- `text/plain`：纯文本正文
- `text/html`：HTML 正文
- `attachment`：真正附件
- `inline`：正文中引用的图片或资源

### 5. 提取 URL

```bash
grep -aoP 'https?://[^"'\''<>() ]+' mail.eml | sort -u
```

如果正文是 HTML 且经过 base64 编码，这条命令可能提取不全，这时建议先把正文解出来。

### 6. 提取邮箱地址

```bash
grep -aoP '[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}' mail.eml | sort -u
```

用途：

- 找额外联系人
- 找隐藏在正文和头部里的域名线索

## MIME 结构怎么看

一个常见邮件结构可能是：

```text
multipart/mixed
  ├─ multipart/related
  │   ├─ text/html
  │   └─ inline image/jpeg
  └─ attachment application/pdf
```

含义：

- 最外层 `mixed`：表示有多个部分，往往是“正文 + 附件”
- `related`：表示正文和它引用的资源放在一起
- `inline`：常见于 logo、签名图片、正文嵌图
- `attachment`：通常是用户能直接下载的附件

如果你看到的是：

```text
multipart/mixed
text/html
image/jpeg inline
image/jpeg attachment
```

那它基本就是：

- 一段 HTML 邮件
- 一张正文内嵌图
- 一张附件图

## 提取附件

### 方法 1：Python 标准库提取，最通用

```bash
mkdir -p eml_out
python3 - <<'PY'
from email import policy
from email.parser import BytesParser
from pathlib import Path

eml = Path("mail.eml")
out = Path("eml_out")
out.mkdir(exist_ok=True)

with eml.open("rb") as f:
    msg = BytesParser(policy=policy.default).parse(f)

idx = 0
for part in msg.walk():
    name = part.get_filename()
    ctype = part.get_content_type()
    disp = part.get_content_disposition()
    payload = part.get_payload(decode=True)

    if name:
        path = out / name
        path.write_bytes(payload or b"")
        print(f"[+] saved: {path} | type={ctype} | disposition={disp}")
    elif payload and ctype not in ("text/plain", "text/html", "multipart/mixed", "multipart/related", "multipart/alternative"):
        idx += 1
        path = out / f"part_{idx}.bin"
        path.write_bytes(payload)
        print(f"[+] saved unnamed: {path} | type={ctype} | disposition={disp}")
PY
```

优点：

- Kali 基本都有 `python3`
- 不依赖额外工具
- 对 `.eml` 很稳

### 方法 2：如果装了 ripmime

```bash
ripmime -i mail.eml -d eml_out
```

### 方法 3：如果装了 munpack

```bash
munpack -f mail.eml
```

## 提取正文

### 提取纯文本/HTML 正文

```bash
python3 - <<'PY'
from email import policy
from email.parser import BytesParser

with open("mail.eml", "rb") as f:
    msg = BytesParser(policy=policy.default).parse(f)

for part in msg.walk():
    ctype = part.get_content_type()
    if ctype in ("text/plain", "text/html"):
        print("=" * 80)
        print(ctype)
        print("=" * 80)
        try:
            print(part.get_content())
        except Exception:
            print(part.get_payload(decode=True))
PY
```

如果正文是 HTML，可以把输出保存成 `.html` 再用浏览器查看源码。

## 附件提取后做什么

提出来后，至少先做这几步：

```bash
file eml_out/*
sha256sum eml_out/*
exiftool eml_out/*
strings -a eml_out/* | less
```

看什么：

- 是否真的是它声称的类型
- 是否有可疑元数据、作者名、生成器、拍摄设备、路径
- 是否藏了 URL、命令、PowerShell、可执行特征、口令提示

如果附件是图片，常见下一步：

```bash
exiftool image.jpg
strings -a image.jpg | less
binwalk image.jpg
zsteg image.png
```

如果附件是 Office/PDF，常见下一步：

```bash
oleid file.doc
pdfid file.pdf
exiftool file.pdf
strings -a file.pdf | less
```

如果附件是压缩包：

```bash
7z l file.zip
binwalk file.zip
```

## HTML 正文里常见可疑点

- 链接显示文字和真实 `href` 不一致
- 域名是仿冒拼写，比如：
  `micros0ft.com`
  `paypaI.com`
  `company-secure-login.com`
- 使用 URL 缩短服务
- 使用大量内嵌样式伪装正规通知
- 有“立即验证”“账户冻结”“紧急付款”“领奖”“投资回报”类社工措辞
- 引导下载压缩包、宏文档、脚本

提取 HTML 里链接：

```bash
python3 - <<'PY'
from email import policy
from email.parser import BytesParser
import re

with open("mail.eml", "rb") as f:
    msg = BytesParser(policy=policy.default).parse(f)

for part in msg.walk():
    if part.get_content_type() == "text/html":
        html = part.get_content()
        for u in sorted(set(re.findall(r'https?://[^"\\'<> )]+', html))):
            print(u)
PY
```

## 常见头字段怎么理解

### From

显示给用户看的发件人。可以伪造，所以不能只看这一项。

### Reply-To

用户点击回复时真正发往的地址。钓鱼邮件很喜欢把它改成别的地址。

### Return-Path

退信地址。能帮助判断真实投递路径，但也不能单独当成绝对可信依据。

### Received

邮件经过的服务器链。越靠下通常越接近源头。

### Authentication-Results

收件服务器对 SPF、DKIM、DMARC 的验证结果。

### Message-Id

邮件唯一标识。可以用来辅助判断是正常客户端发送，还是脚本、群发器、伪造工具生成。

## 比赛或实战里的典型判断点

### 1. 发件人伪装

比如：

- `From: finance@company.com`
- `Reply-To: somebody123@gmail.com`

这通常就不正常。

### 2. 认证不一致

比如：

- 显示是某公司域名
- 但 `spf=fail`、`dkim=fail`

需要高度怀疑伪造。

### 3. 附件和主题强相关

诈骗邮件常见：

- 发票
- 汇款通知
- 合同
- 付款凭证
- 简历
- 投资邀请

### 4. 来源 IP 可疑

如果最初来源是：

- 家宽 IP
- 动态 IP
- 与宣称公司所在地完全不符

通常要重点看。

## 常见坑

- `From` 看起来正常，不代表邮件就正常。
- 只有 HTML 正文，不代表没有恶意行为，钓鱼往往就在链接里。
- `inline` 图片不一定是无害的，有时会带跟踪、诱导或藏线索。
- `binwalk` 不是邮件 MIME 解析器，分析 `.eml` 时通常不是首选。
- 邮件里的 URL 可能经过跳转、缩短、HTML 编码或 base64 包裹。
- 一个附件名看着像图片、PDF，不代表内容真的就是那个类型。

## 和其他工具怎么配合

- `pdfid`：分析 PDF 附件
- `exiftool`：分析图片、PDF、Office 元数据
- `binwalk`：分析提取出来的附件或图片是否嵌套别的内容
- `yara`：对附件和正文落地内容做规则匹配
- `strings`：从附件中找链接、命令、敏感词
- `sha256sum`：给 IOC 留指纹

## 简短口诀

可以记成：

```text
先看头 -> 再看 MIME -> 提附件 -> 查链接 -> 验附件 -> 串时间线
```

如果只记最常用的 4 组动作，通常就是：

```bash
grep -nE '^(From|Reply-To|Return-Path|Authentication-Results|Received):' mail.eml
grep -nE 'Content-Type:|Content-Disposition:|filename=|name=' mail.eml
python3 提取附件
file / exiftool / strings 分析附件
```
