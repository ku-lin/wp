# 服务器取证总结

来源：<https://forensics.xidian.edu.cn/wiki/writeups/Pinghang2026/>

适用范围：本总结只整理该题解中和服务器取证直接相关的部分，主要对应 `API 服务器`、`入侵分析`，并吸收少量和服务器投毒验证有关的主机侧证据。

## 1. 这套题里涉及到的服务器取证知识点

### 1.1 Linux 服务器基础信息取证

- 内核版本既可以直接在取证工具里看，也可以仿真后用 `uname -r` 获取。
- 登录次数不要只看单一日志源，`last`、登录日志分析结果、用户列表统计结果需要交叉验证。
- 崩溃（Crash）场景可能导致常规登录日志少一条，因此题里出现了“登录日志里 9 次，实际 10 次”的情况。

### 1.2 服务配置与源码取证

- Redis 服务密码可直接从配置文件 `/etc/redis/redis.conf` 中提取。
- API 服务源码路径本身就是核心证据点，本题中路径为 `/home/zaoqiwang/claude-relay-service`。
- 后台初始化数据、管理员账号、密码哈希算法等，通常可以从类似 `data/init.json` 的初始化文件中提取。
- 通过哈希前缀可以快速判断加密算法，本题管理员密码哈希前缀为 `$argon2id$`，对应 `argon2id`。

### 1.3 密码恢复与后台登录

- 题目给了密码模式 `b1?????b`，这类场景既可以从字典里筛符合模式的候选，也可以直接用掩码爆破。
- `HashCat` 在这种“已知前后缀+中间为数字”的口令上很高效。
- 后台信息不一定非要“正常启动前端”才能看到，管理后台、Redis、日志、导出数据都可能是旁路取证入口。

### 1.4 Redis 数据取证

- Webhook/通知配置可以直接 `GET` 某个键取值。
- 统计类数据可以通过 `HGETALL` 读取 hash。
- 时间排序索引常见于 `ZSET`，时间戳可通过 `ZRANGE ... WITHSCORES` 提取后再转 UTC 时间。
- 这类题的关键不是“会不会点后台”，而是知道后台展示的数据通常都能在 Redis 原始键值里找到。

### 1.5 服务端投毒逻辑分析

- 先从源码里全局搜索关键函数名，本题关键入口是 `inject_bash_blocks`。
- 如果 JS 只是壳，真正逻辑在 wasm 中，需要继续跟到 `bash_block_injector_bg.wasm`。
- wasm 可以用 `wasm2wat` 反编译，也可以转 C 后再编译进 IDA 分析。
- 对外暴露函数时，除了静态逆向，还可以直接写测试脚本调用导出函数，做“黑盒+白盒结合”的行为验证。

### 1.6 服务器恶意逻辑参数求解

- `should_inject_for_ua(ua, ip)` 这类函数，适合用脚本批量构造 UA 和 IP 验证。
- 本题的关键逻辑包括三层：
  - UA 关键词过滤
  - 同 IP 短时间窗口判断
  - 满足前两者后还存在概率门
- 这种题本质是在做“服务端恶意触发条件取证”，重点是还原触发机制，而不只是拿答案。

### 1.7 域渗透/BloodHound 证据分析

- 域控提权题不一定看 Neo4j 库本身，题解明确说明本题真正相关的是 U 盘里的 `BloodHound.zip` 数据。
- 目录复制相关权限：
  - `GetChanges`
  - `GetChangesAll`
  - `GetChangesInFilteredSet`
- 这三类权限组合起来，可以推出 `DCSync` 能力。
- 攻击路径题的核心是把“已控用户 -> 可控主机 -> 域对象”的边串起来，恢复完整提权链。

### 1.8 主机侧反查服务器投毒

- 服务器投毒不只在服务器端留痕，受害者主机上也可能留有执行许可记录。
- 本题里可在 `.claude/settings.local.json` 中看到允许执行的恶意命令，可反向印证服务器投毒 payload。

## 2. 这类题怎么取证

### 2.1 先做基础画像

1. 挂载或仿真服务器镜像。
2. 提取系统基础信息：
   - 内核版本
   - 用户登录情况
   - 服务进程/目录
3. 记录所有核心服务路径、配置文件路径、数据库路径。

### 2.2 再做配置与凭据取证

1. 看配置文件：
   - `/etc/redis/redis.conf`
   - 服务源码目录下的配置和初始化文件
2. 提取账号、密码哈希、算法标识。
3. 根据题目给定模式决定是字典爆破还是掩码爆破。

### 2.3 再做后台与数据层取证

1. 能登录后台就登录后台取证。
2. 前端有问题时，直接查 Redis。
3. 本题典型证据点：
   - `GET webhook_config:default`
   - `HGETALL usage:global:total`
   - `ZRANGE apikey:idx:createdAt 0 -1 WITHSCORES`
4. 把界面数据和底层原始键值对应起来，确认取证链闭环。

### 2.4 恶意投毒逻辑取证

1. 在源码中全局搜索关键函数，如：
   - `inject_bash_blocks`
   - `should_inject_for_ua`
2. 看 JS 是否只是包装层。
3. 若真实逻辑在 wasm：
   - 抽出 wasm
   - 反编译为 wat/c
   - 结合 IDA/大模型辅助阅读
4. 同时编写调用脚本，直接跑函数做行为验证。
5. 还原：
   - 实际注入 payload
   - 哪些 UA 能触发
   - 时间窗口阈值
   - 触发概率

### 2.5 域控提权链取证

1. 优先确认真正证据源是不是 BloodHound 导出包，而不是数据库运行环境。
2. 查对象之间的关系边：
   - `MemberOf`
   - `GenericAll`
   - `GetChanges*`
3. 判断中间主机是否具备 DC 身份或目录复制权限。
4. 最后把攻击路径写成完整链路，而不是只写终点。

## 5. 这套题暴露出的取证思路

- 服务器题不要只盯着“系统日志”，源码、配置、Redis、后台、导出数据都同样重要。
- 登录次数、时间戳、统计量这类题要做多源交叉验证，不能只信一个视图。
- 遇到 wasm/混淆逻辑时，不要死磕纯静态；写脚本调用原函数，经常更快。
- 域渗透题里，图数据库软件本身不一定是重点，真正重点是导出的关系数据。
- 主机侧和服务器侧应当互相印证，尤其是投毒、C2、恶意命令这类题。

## 6. 易错点

- 登录日志可能漏掉崩溃前后的记录，导致次数偏少。
- 后台前端跑不起来，不代表拿不到后台数据；Redis 往往更直接。
- Neo4j 密码不等于域渗透题真正证据源，本题真正有用的是 BloodHound 导出数据。
- 只做静态逆向、不做函数级调用验证，容易错过概率门和时间窗这类动态条件。
