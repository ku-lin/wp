## 前言

在 Linux 中文件操作大致可以分成两层：

- C 标准库提供的库函数：`fopen`、`fclose`、`fwrite`、`fread`
- 操作系统提供的系统调用：`open`、`close`、`write`、`read`

可以把它们理解成“上下层关系”：

- 系统调用更靠近内核
- 库函数在用户态进行了进一步封装
- 很多库函数最终还是会调用底层系统调用完成实际工作

---

## 一、文件描述符 fd

### 1. fd 是什么

文件描述符（`file descriptor`，简称 `fd`）本质上是一个非负整数，用来标识进程当前打开的某个文件对象。

在 Linux 中，进程默认会打开三个标准文件描述符：

- `0`：标准输入 `stdin`
- `1`：标准输出 `stdout`
- `2`：标准错误 `stderr`

通常可以简单理解为：

- `0` 对应键盘输入
- `1` 对应终端输出
- `2` 对应终端错误输出

### 2. fd 为什么能代表文件

当进程打开一个文件时，操作系统会在内核中创建对应的数据结构来描述这个文件。

对进程来说，进程控制块中会维护一张“已打开文件表”。文件描述符就是这张表中的索引值，也就是数组下标。通过这个下标，进程就可以找到对应的文件对象并进行读写。

可以把它理解成：

- 内核真正管理的是文件对象
- 进程手里拿到的是一个整数编号
- 这个整数编号就是 `fd`

### 3. fd 的分配规则

Linux 下打开文件时，通常会优先分配“当前可用的最小文件描述符”。

因为系统启动时已经占用了：

- `0`
- `1`
- `2`

所以第一次 `open` 新文件时，常见返回值就是 `3`。

如果某个较小的 fd 被关闭了，那么下次打开文件时，系统会优先复用它。

例如：

1. 如果关闭了 `0`
2. 再连续打开两个文件
3. 第一个新文件可能拿到 `0`
4. 第二个新文件可能拿到 `3`

这也是很多 I/O 重定向和题目利用里常用的基础知识。

## 二、FILE 结构体

`FILE` 是 C 标准库中用于描述文件流的数据结构，常见于：

```c
FILE *fp = fopen("test.txt", "r");
```

从使用角度看，`fd` 和 `FILE *` 都能用于文件操作，但它们并不在同一层：

- `fd` 是内核接口层面的“文件编号”
- `FILE *` 是 C 标准库层面的“文件流对象”

### 1. FILE 结构体里关注什么

原文重点强调了 `FILE` 中两个非常重要的方面：

- 文件描述符
- 缓冲区

### 2. 文件描述符

很多标准 I/O 函数虽然表面上操作的是 `FILE *`，但这个结构内部通常仍然关联着一个底层 `fd`。

也就是说：

- `fopen` 得到的是 `FILE *`
- 但它底层往往仍然会对应一个文件描述符

### 3. 缓冲区

`FILE` 相比单纯的系统调用，一个非常重要的特点就是“标准库缓冲”。

这也是 `printf` / `fwrite` 和 `write` 行为差异的关键。

---

## 三、缓冲区示例

下面是一个典型示例：

```c
#include <stdio.h>
#include <string.h>
#include <unistd.h>

int main(void)
{
    const char *msg1 = "beautiful gril printf\n";
    const char *msg2 = "beautiful gril fwrite\n";
    const char *msg3 = "beautiful gril write\n";

    printf("%s", msg1);
    fwrite(msg2, strlen(msg2), 1, stdout);
    write(1, msg3, strlen(msg3));

    fork();
    return 0;
}
```

### 1. 现象

在某些运行环境下，如果把输出重定向到普通文件，会发现：

- `printf` 的内容可能出现两次
- `fwrite` 的内容可能出现两次
- `write` 的内容通常只出现一次

### 2. 原因分析

原因在于库函数和系统调用对缓冲区的处理不同。

常见缓冲策略可以概括为：

- 无缓冲
- 行缓冲
- 全缓冲

对标准 I/O 来说：

- 输出到终端时，很多场景下是“行缓冲”
- 输出重定向到普通文件时，常常变成“全缓冲”

这会导致：

- `printf` 和 `fwrite` 的数据先进入用户态缓冲区
- 调用 `fork()` 时，父进程的用户态内存会被子进程继承
- 如果缓冲区中的数据当时还没刷新出去，那么父子进程后续都可能各自刷新一次
- 结果就是文件里看到两份相同内容

而 `write` 是系统调用，通常直接进入内核处理，不依赖 C 标准库缓冲区，所以不会因为这个原因重复输出。

### 3. 这个现象的核心结论

可以把结论记成一句话：

- `printf` / `fwrite` 依赖 `FILE` 缓冲区
- `write` 直接走系统调用，通常不受标准库缓冲影响

## 四、fd 和 FILE * 的区别

两者可以从下面几个角度理解：


| 对比项           | `fd`                    | `FILE *`                    |
| ---------------- | ----------------------- | --------------------------- |
| 所属层次         | 内核/系统调用接口       | C 标准库                    |
| 类型             | 非负整数                | 指向结构体的指针            |
| 典型函数         | `open/read/write/close` | `fopen/fread/fwrite/fclose` |
| 是否带标准库缓冲 | 否                      | 是                          |
| 使用粒度         | 更底层                  | 更方便、更高级              |

### 简单理解

- 想直接和内核接口打交道，用 `fd`
- 想用标准库提供的易用接口，用 `FILE *`

---

## 五、实际开发中的理解方式

在学习 Linux I/O 时，可以这样建立概念：

1. `open` 返回的是 `fd`
2. `fd` 是进程打开文件表中的索引
3. `fopen` 返回的是 `FILE *`
4. `FILE *` 是标准库包装后的流对象
5. `FILE *` 底层通常仍然关联某个 `fd`
6. 标准库 I/O 和系统调用 I/O 的重要差别之一，就是缓冲机制

---

## 六、总结

本文最核心的几个点如下：

- 文件描述符 `fd` 是一个非负整数，用来标识进程打开的文件
- Linux 默认已经打开了 `0`、`1`、`2` 三个标准文件描述符
- 新打开文件时，系统倾向于分配当前最小可用 fd
- `FILE *` 是 C 标准库中的文件流对象，不等同于 fd，但通常会关联底层 fd
- `printf`、`fwrite` 这类库函数通常带缓冲区
- `write` 这类系统调用更底层，通常不经过标准库缓冲
- `fork()` 之后如果缓冲区内容尚未刷新，可能导致标准 I/O 输出重复
