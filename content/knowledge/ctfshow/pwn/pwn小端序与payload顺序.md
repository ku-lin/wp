做 pwn 时经常会遇到一种错觉：

```python
payload += p32(0x11)
payload += p32(0x0)
```

明明 payload 里先写了 `0x11`，但在 GDB 里看起来好像 `0x11` 跑到后面去了。

这个问题的核心不是 `p32` 写反了，而是：

```text
内存按地址从低到高排列字节
整数显示时按数值从高位到低位显示
```

小端序就是把一个整数的低字节放在低地址。

## 1. 先记住一句话

小端序：

```text
低地址放低字节
高地址放高字节
```

比如：

```python
p32(0x12345678)
```

真实写进内存的字节是：

```text
78 56 34 12
```

如果起始地址是 `0x804a000`：

```text
地址          字节
0x804a000     78
0x804a001     56
0x804a002     34
0x804a003     12
```

但是 GDB 如果按 4 字节整数看：

```gdb
x/wx 0x804a000
```

会显示：

```text
0x12345678
```

这不是内存字节顺序变了，而是 GDB 把这 4 个字节解释成了一个整数。

## 2. p32 和 p64 到底写了什么

`p32(x)` 是把一个 32 位整数按小端序打包成 4 个字节。

```python
p32(0x11)
```

写入的是：

```text
11 00 00 00
```

`p64(x)` 是把一个 64 位整数按小端序打包成 8 个字节。

```python
p64(0x11)
```

写入的是：

```text
11 00 00 00 00 00 00 00
```

所以：

```python
p64(0x11)
```

等价于：

```python
p32(0x11) + p32(0x0)
```

前提是你想要的内存布局确实是：

```text
低地址: 0x00000011
高地址: 0x00000000
```

## 3. 用 pwn112 的例子理解

题目里关键判断类似：

```asm
lea eax, [global_buf + 0x34]
mov edx, [eax+0x4]   ; 取 global_buf+0x38 的 4 字节
mov eax, [eax]       ; 取 global_buf+0x34 的 4 字节
mov ecx, eax
xor ecx, 0x11
or eax, ecx
test eax, eax
```

想让分支成立，需要满足：

```text
*(uint32_t *)(global_buf + 0x34) ^ 0x11 == 0
*(uint32_t *)(global_buf + 0x38) == 0
```

也就是：

```text
global_buf+0x34: 0x00000011
global_buf+0x38: 0x00000000
```

payload 应该写：

```python
payload = b"A" * 0x34
payload += p32(0x11)
payload += p32(0x0)
```

也可以写成：

```python
payload = b"A" * 0x34
payload += p64(0x11)
```

真实内存字节是：

```text
global_buf+0x34: 11 00 00 00
global_buf+0x38: 00 00 00 00
```

按 32 位整数看：

```text
global_buf+0x34: 0x00000011
global_buf+0x38: 0x00000000
```

按 64 位整数看：

```text
global_buf+0x34: 0x0000000000000011
```

这三种显示方式说的是同一块内存。

## 4. GDB 里怎么避免看错

如果你想看真实字节顺序，用：

```gdb
x/8xb global_buf+0x34
```

输出类似：

```text
0x....: 0x11 0x00 0x00 0x00 0x00 0x00 0x00 0x00
```

这里就是内存地址从低到高的真实字节。

如果你用：

```gdb
x/2wx global_buf+0x34
```

GDB 会每 4 字节解释成一个整数：

```text
0x....: 0x00000011  0x00000000
```

如果你用：

```gdb
x/gx global_buf+0x34
```

GDB 会把 8 字节解释成一个 64 位整数：

```text
0x....: 0x0000000000000011
```

容易混乱的点是：`x/gx` 右边这个数字不是按字节地址顺序打印的，它是在打印一个数值。

## 5. 三种视角不要混

同一段内存：

```text
11 00 00 00 00 00 00 00
```

可以有三种常见解释：

```text
按字节看:       11 00 00 00 00 00 00 00
按两个 dword 看: 0x00000011  0x00000000
按一个 qword 看: 0x0000000000000011
```

它们不是互相矛盾，而是观察方式不同。

## 6. 做 payload 时按地址顺序写

payload 是从低地址往高地址写进去的。

比如：

```python
payload += p32(0x11111111)
payload += p32(0x22222222)
payload += p32(0x33333333)
```

内存布局就是：

```text
低地址 -> 高地址
11 11 11 11  22 22 22 22  33 33 33 33
```

按 32 位整数看：

```text
0x11111111  0x22222222  0x33333333
```

所以构造 payload 时，先想清楚目标地址：

```text
addr+0x00 应该是什么
addr+0x04 应该是什么
addr+0x08 应该是什么
```

然后按这个顺序写：

```python
payload += p32(value_at_addr_0x00)
payload += p32(value_at_addr_0x04)
payload += p32(value_at_addr_0x08)
```

## 7. 什么时候能用 p64 代替两个 p32

可以用：

```python
p64(0x11)
```

代替：

```python
p32(0x11) + p32(0x0)
```

因为两者字节完全一样：

```text
11 00 00 00 00 00 00 00
```

但是不要无脑用 `p64`。如果目标是两个独立的 32 位字段，要先确认高 4 字节确实应该是 0。

例如你想写：

```text
addr+0x00: 0x11111111
addr+0x04: 0x22222222
```

那应该写：

```python
payload += p32(0x11111111)
payload += p32(0x22222222)
```

或者写成一个 64 位数：

```python
payload += p64(0x2222222211111111)
```

注意这里看起来是 `0x22222222` 在前，`0x11111111` 在后。

原因是这个 64 位整数的小端字节为：

```text
11 11 11 11 22 22 22 22
```

所以如果你不熟，写两个 `p32` 更不容易错。

## 8. 32 位程序里能不能用 p64

能用。`p64` 只是打包 8 个字节，不代表程序必须是 64 位。

例如 32 位程序里有两个连续的 4 字节字段：

```text
field1 = 0x00000011
field2 = 0x00000000
```

可以写：

```python
payload += p64(0x11)
```

但是 32 位 ROP 里返回地址、函数参数、GOT 地址通常还是 4 字节，应该用 `p32`。

建议：

```python
context(os="linux", arch="i386")
```

然后明确使用：

```python
p32(addr)
```

只有在你确实要一次写 8 字节布局时，再用 `p64`。

## 9. 常见错误

错误 1：把 GDB 的数值显示当成内存字节顺序。

```gdb
x/gx addr
```

看到：

```text
0x0000000000000011
```

就以为 `0x11` 在最后。其实真实字节可能是：

```text
11 00 00 00 00 00 00 00
```

想确认真实字节，用：

```gdb
x/8xb addr
```

错误 2：把 `p64(0x2222222211111111)` 写成 `p64(0x1111111122222222)`。

如果目标内存是：

```text
addr+0x00: 0x11111111
addr+0x04: 0x22222222
```

正确的 64 位整数是：

```python
p64(0x2222222211111111)
```

不熟的时候直接写：

```python
p32(0x11111111) + p32(0x22222222)
```

更稳。

错误 3：忘记 payload 是从低地址往高地址覆盖。

比如：

```python
payload = b"A" * offset
payload += p32(fake_ebp)
payload += p32(ret_addr)
```

内存里一定是：

```text
offset+0x00: fake_ebp
offset+0x04: ret_addr
```

不要被 `ret_addr` 更重要这件事影响顺序。

## 10. 实战检查清单

写 payload 前问自己：

```text
1. 目标字段在什么地址？
2. 这个字段是 1 字节、4 字节还是 8 字节？
3. 我是想看真实字节，还是想看整数值？
4. 如果是连续字段，每个字段在低地址到高地址的顺序是什么？
5. p32/p64 打包出来的字节是不是和目标内存布局一致？
```

调试时优先用：

```gdb
x/16xb addr
```

确认真实字节。

再用：

```gdb
x/4wx addr
```

确认 32 位字段。

最后需要看 64 位整体时再用：

```gdb
x/gx addr
```

## 11. 最后再背一次

```text
p32(0x12345678) -> 78 56 34 12
p64(0x1122334455667788) -> 88 77 66 55 44 33 22 11
```

payload 的拼接顺序就是内存从低地址到高地址的覆盖顺序。

只要你脑子里先画出：

```text
低地址 -> 高地址
```

再决定每一格填什么，小端序就不会那么容易把你拐跑。
