## 1. 这是什么软件

`Exeinfo PE` 是一个 **Windows PE 可执行文件快速识别工具**，常用于：

- 判断程序是不是 `EXE`、`DLL`、`SYS`
- 判断是 `32 位` 还是 `64 位`
- 粗略识别是否 **加壳 / 压缩 / 保护**
- 查看入口点、区段、链接器版本、子系统等基础 PE 信息
- 给逆向分析人员一个“先看一眼”的快速结论

它的特点是 **上手快、看样本快**，适合放在静态分析的第一步使用。  
但它不是权威判定工具，很多结论属于 **特征匹配或启发式判断**，最后仍要结合 `PE-bear`、`DIE`、`IDA`、`x64dbg`、`Detect It Easy` 等工具进一步确认。

### 2.3 入口点文件偏移是 `0x4C80`

截图中：

- `File Offset : 00004C80`

说明：

- 如果你用十六进制编辑器看文件内容，可以直接跳到文件偏移 `0x4C80`

---

### 2.4 入口点前几个字节像正常 64 位函数开头

截图中：

- `First Bytes : 48 83 EC 28 E8`

这串字节常见于 64 位程序入口附近，含义大致是：

- `48 83 EC 28`：调整栈空间
- `E8`：接下来通常是一次 `call`

这通常说明入口点看起来像 **真实代码入口**，不像明显的壳跳板。

截图中：

- `File Size : 00433000h`

换算后约为：

- `4,403,200` 字节

这个体积不算特别小，结合后面的资源信息，说明它可能带有较多资源或内嵌内容。

---

### 2.6 链接器版本是 `14.35`

截图中：

- `Linker Info : 14.35`

这通常表示：

- 程序大概率使用了 **较新的 Microsoft Visual C++ / Visual Studio 工具链**

注意：

- 这不是“编译时间”
- 只是说明它大概率由哪个时代的工具链生成

---

### 2.7 没有 Overlay 叠加数据

截图中：

- `Overlay : NO 00000000`

说明：

- 文件末尾没有明显追加一段“壳数据”或“额外附加内容”
- 至少从 overlay 这个角度看，不像那种把主数据直接拼接到文件尾的简单封装方式

---

### 2.8 资源区很大，且样本看起来没有加壳

截图下方两条信息很重要：

- `Big sec. 06 [.rsrc]`
- `Not packed , try debug www.ollydbg.de or www.x64dbg...`

大意是：

- 这个 PE 一共有大约 `6` 个区段
- 其中 `.rsrc` 资源段比较大
- `Exeinfo PE` 的判断是：**不像常见加壳样本**

这在实战里很有价值，因为它提示你：

- 可以优先做 **静态分析**
- 关注 `.rsrc` 段里是否藏了图片、配置、公钥、压缩数据、第二阶段载荷

如果这是题目里的恶意程序，这种“大资源段 + 未加壳”的组合，常常意味着：

- 恶意配置在资源里
- 释放文件在资源里
- 壁纸、勒索信、公钥、内嵌 payload 在资源里

---

### 2.9 中间那行识别串是“启发式标签”，只能参考

截图中类似：

`22 - (exe 4883EC28-48) [Deb] - no sec. Cab.7z.Zip [Win Vista] [Deb. 13]`

这类内容是 `Exeinfo PE` 根据特征库给出的 **识别提示**，不是严格标准结论。  
通常可以把它理解成：

- 入口点字节特征被识别到了
- 样本更像正常可执行程序，不像典型壳入口
- 可能带有调试/普通编译特征

但这一行不要单独作为证据使用，最好只拿它做方向提示。

## 3. 这张图对你逆向分析意味着什么

如果你是在做题或分析恶意程序，这张图已经给了你一个很明确的起手方向：

1. 这是 **64 位 GUI 程序**
2. **入口点比较正常**
3. **不像常见加壳样本**
4. **`.rsrc` 资源段很大，值得重点检查**
5. 适合直接上 `IDA64`、`x64dbg`、`PE-bear`、`Resource Hacker` 做进一步分析

也就是说，下一步最合理的动作通常不是“先脱壳”，而是：

1. 用 `IDA64` 看字符串、导入表、资源释放函数
2. 用 `PE-bear` 或 `Resource Hacker` 检查 `.rsrc`
3. 看有没有内嵌图片、公钥、配置、压缩包、第二个 EXE
4. 必要时在 `x64dbg` 里从入口点跟一下执行流程

## 4. Exeinfo PE 怎么用

## 4.1 基本用法

1. 打开 `Exeinfo PE`
2. 将目标文件拖进去，或者点击左上角文件选择按钮
3. 观察以下核心字段：
   - `Entry Point`
   - `EP Section`
   - `File Offset`
   - `Linker Info`
   - `SubSystem`
   - `Overlay`
   - 底部识别结果
4. 根据结果决定后续工具链

---

## 4.2 逆向时优先看什么

建议先看这几项：

- `Image is 64 bit executable`
  - 决定你应该优先用 `IDA64` 还是 `IDA`
- `SubSystem`
  - 判断是图形程序还是控制台程序
- `EP Section`
  - 看入口是否在正常代码段
- `First Bytes`
  - 粗略判断入口是否像正常代码或壳跳板
- `Overlay`
  - 判断文件尾是否有附加数据
- 底部 `Not packed / Packed`
  - 快速判断是否可能加壳

---

## 4.3 常见分析思路

### 场景 1：显示 `Not packed`

可以优先：

- 丢进 `IDA` 看字符串
- 看导入函数
- 看资源段
- 动调跟入口点

### 场景 2：显示 `Packed`

可以优先：

- 用 `DIE`、`PEiD` 交叉确认壳类型
- 尝试脱壳
- 在调试器里找 OEP

### 场景 3：`.rsrc` 很大

优先怀疑：

- 内嵌图片
- 内嵌压缩包
- 配置文件
- 公钥/证书
- 第二阶段载荷

---

## 5. Exeinfo PE 的局限

这个工具很方便，但要注意：

- 它的很多结果是 **特征匹配**
- `Not packed` 不等于一定没壳
- `Packed` 也不等于一定就是那个壳
- 链接器版本不能直接等同于真实开发环境
- 编译时间、签名、资源、导入表等还需要其他工具复核

所以正确姿势是：

- 用它做 **快速预判**
- 不要把它当最终证据

## 7. 推荐搭配工具

- `IDA / IDA64`：看伪代码、字符串、交叉引用
- `x64dbg`：动态调试
- `PE-bear`：看节表、导入表、资源、overlay
- `Detect It Easy (DIE)`：交叉识别编译器、壳、保护
- `Resource Hacker`：查看和导出资源
- `HxD`：直接看十六进制和文件偏移

---

## 8. 适合你当前样本的实战建议

如果你现在分析的是 `LOVE2.exe`，建议按这个顺序继续：

1. 用 `IDA64` 打开
2. 先看 `Strings`
3. 再看导入函数，尤其是：
   - 网络通信

   - 资源释放相关
4. 检查 `.rsrc` 是否有内嵌文件
5. 必要时用 `x64dbg` 从入口点跟踪
